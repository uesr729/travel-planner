# AI 旅行规划师 — 测试策略

## 1. 项目代码分析

### 1.1 项目整体结构

| 目录/文件 | 作用 |
|-----------|------|
| `travel_planner/__init__.py` | 应用工厂，初始化 Flask + SQLAlchemy |
| `travel_planner/models.py` | Itinerary 数据模型 + to_dict() 方法 |
| `travel_planner/routes/main.py` | 5 条路由：首页/生成/查看/分享/历史 |
| `travel_planner/services/llm_service.py` | LLM 行程生成（DeepSeek API + Mock 双模式） |
| `travel_planner/services/mock_data.py` | 内置 20+ 城市坐标 + 5 城市详细数据 |
| `travel_planner/services/weather.py` | 天气查询（OpenWeather API + Mock 双模式） |
| `travel_planner/templates/` | 6 个 Jinja2 模板 |
| `travel_planner/static/` | CSS 样式和 JS 交互代码 |

### 1.2 模块依赖关系

```
mock_data.py（无依赖）
    ↑
weather.py → 依赖 config（API Key）
    ↑
llm_service.py → 依赖 mock_data.py、config
    ↑
routes/main.py → 依赖 models、llm_service、weather
    ↑
__init__.py / models.py（最底层）
```

### 1.3 关键数据流

```
前端表单 → POST /generate → llm_service.generate_itinerary()
                                → 尝试 DeepSeek API
                                → 失败则 mock_data.generate_mock_itinerary()
                             → weather.get_weather()
                                → 尝试 OpenWeather API
                                → 失败则 mock_data.generate_mock_weather()
                             → 写入 SQLite（Itinerary.raw_data）
                             → 重定向到 /itinerary/<id>
                             → to_dict() 解析 raw_data JSON
                             → Jinja2 渲染 itinerary.html
```

## 2. 测试策略

### 2.1 模块优先级

| 优先级 | 模块 | 理由 | 测试类型 |
|--------|------|------|---------|
| **P0（核心）** | `routes/main.py` + `models.py` | 核心业务逻辑，用户交互入口 | 集成测试 + 单元测试 |
| **P1（重要）** | `services/mock_data.py` | 核心兜底机制，无 API Key 时唯一数据源 | 单元测试 |
| **P2（一般）** | `services/llm_service.py` | 依赖外部 API，重点测 Mock 回退路径 | 单元测试 |
| **P3（次要）** | `services/weather.py` | 天气是辅助信息，不影响主流程 | 轻量测试 |

### 2.2 测试类型说明

| 测试类型 | 覆盖范围 | 说明 |
|---------|---------|------|
| **单元测试** | models.py, mock_data.py, llm_service.py | 独立测试每个函数/方法的输入输出 |
| **集成测试** | routes/main.py + 完整请求链路 | 测试 HTTP 请求到响应的完整流程 |

### 2.3 需要特别关注的边界情况

| 场景 | 说明 |
|------|------|
| 空目的地提交 | 验证重定向到首页 |
| 非法 days 值 | 负数、0、超界 |
| 非法 budget 值 | 负数、0、极小值 |
| raw_data 损坏 | JSON 解析失败，验证 to_dict 不崩溃 |
| 目的地不在 CITY_DATA 中 | 验证回退到北京数据 |
| 空偏好 | 验证不崩溃 |
| 无效分享令牌 | 验证返回 404 |
| 不存在的 itinerary ID | 验证返回 404 |
| 历史记录为空 | 验证显示空状态引导 |
| Mock 数据生成 | 生成数据是否包含所有必需字段 |

### 2.4 测试执行顺序

```
1. 测试框架配置验证
2. mock_data 单元测试（P1）
3. models 单元测试（P0）
4. llm_service 单元测试（P2）
5. routes 集成测试（P0）→ 包含正常/异常/边界场景
6. 最终全量测试验证
```
